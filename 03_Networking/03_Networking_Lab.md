## 🎯 Мета роботи

Навчитись орієнтуватись у мережевих налаштуваннях віртуальної машини, переглядати IP-адреси та відкриті порти, розуміти критичну різницю між прив'язкою сервісу до `127.0.0.1` та `0.0.0.0`, а також налаштовувати Port Forwarding (прокидання портів) через Vagrantfile та базовий захист через фаєрвол UFW.

> **Контекст:** Ми підготуємо мережу так, щоб у майбутньому отримати доступ до інтерфейсу нашого AI-агента OpenClaw (порт 18789), але зберегти його захищеним від зовнішнього світу.

## 🛠 Покрокова інструкція

### Крок 1: Запускаємо VM та перевіряємо мережу

Підключіться до вашої віртуальної машини, як ви це робили у Темі 2:

```bash
# На хості:
cd devops-course/DevOps/devops-ai-assistant/10_Implementation/01_vm/
vagrant up
vagrant ssh
```

Тепер, знаходячись всередині VM, перевіримо її мережеві інтерфейси:

```bash
# Детальний вивід всіх адаптерів та їхніх IP
ip addr

# Швидкий перегляд тільки IP-адрес
hostname -I
```

**Очікуваний результат:** У виводі `ip addr` ви маєте знайти як мінімум:

- Інтерфейс `lo` (loopback) з адресою `127.0.0.1`.
- Інтерфейс типу `eth0` або `enp0s3` (це віртуальний кабель до вашого хоста через VirtualBox NAT) з адресою `10.0.2.15`.
- Інтерфейс з адресою `192.168.56.10` (private_network з Vagrantfile).

### Крок 2: Хто слухає порти просто зараз?

Перш ніж створювати свої сервіси, подивимось, що вже "відкрито" на чистій Ubuntu:

```bash
# Показати всі TCP-порти, які зараз прослуховуються, і які процеси їх тримають
sudo ss -tlnp
```

**Очікуваний результат:**

```text
State   Recv-Q   Send-Q   Local Address:Port    Peer Address:Port   Process
LISTEN  0        128            0.0.0.0:22           0.0.0.0:*      users:(("sshd",pid=605,fd=3))
```

Ви бачите, що процес `sshd` (SSH-сервер) слухає порт `22` на адресі `0.0.0.0` (тобто на всіх інтерфейсах, приймаючи всі з'єднання).

> 💡 Якщо ви спробуєте виконати стародавню команду `netstat -tlnp`, система повідомить, що вона не знайдена і запропонує `sudo apt install net-tools`. Хоч ми встановили `net-tools` у процесі provisioning-у, у світі сучасного Linux звикайте використовувати `ss`.

### Крок 3: Симуляція агента — прив'язка до 0.0.0.0

Щоб зрозуміти роботу мережі, ми не будемо зараз розгортати всього OpenClaw агента (це довго). Ми використаємо Python для створення короткоживучого "фейкового" веб-сервера.

```bash
# Запускаємо простий веб-сервер на порту 18789 ЯВНО кажучи йому слухати на всіх адресах
# Символ & в кінці запускає процес у "фоновому" режимі
python3 -m http.server 18789 --bind 0.0.0.0 &
```

Вивід покаже: `Serving HTTP on 0.0.0.0 port 18789 (http://0.0.0.0:18789/) ...`

Тепер перевірте порти знову:

```bash
sudo ss -tlnp | grep 18789
# Очікуваний результат: LISTEN ... 0.0.0.0:18789 ... python3
```

Спробуйте "достукатись" до цього сервера зсередини VM:

```bash
# Перевірка локально (має повернути HTML-код поточної теки)
curl localhost:18789
```

**Очікуваний результат:** Успішна відповідь, ви побачите сторінку з лістингом директорії.

Тепер відкрийте браузер **на вашому комп'ютері (хості)** і спробуйте перейти за адресою:
`http://127.0.0.1:18789`

**Очікуваний результат у браузері:** "Не вдається отримати доступ до сайту" (Connection Refused).

**Чому так?** Тому що `127.0.0.1` у браузері вашого ноутбука — це *сам ваш ноутбук*, а наш сервер запущений всередині ВМ!

**Заходимо правильно (через Private Network):**
У базових налаштуваннях через `Vagrantfile` ми створили пряму локальну мережу між ноутбуком і ВМ. Її адреса на стороні ВМ — `192.168.56.10`.
Отже, просто зараз зайдіть у браузері хоста на:
`http://192.168.56.10:18789`

**Очікуваний результат:** Сайт відразу відкриється! Проста і надійна локальна розробка, без зайвих "закопувань" у додаткове прокидання портів (Port Forwarding), оскільки у нас вже є прямий зв'язок.

**ПРОБЛЕМА БЕЗПЕКИ:** Оскільки зараз наш сервер слухає `0.0.0.0` (на всіх інтерфейсах), до нього можна просто так зайти. Можливо, для тестового вебсайту це нормально, але наш AI-агент міститиме фінансові та медичні дані. Його треба прив'язати тільки до локального інтерфейсу ОС!

### Крок 4: Правильна ізоляція агента (прив'язка до 127.0.0.1)

Зупинимо наш відкритий сервер:

```bash
# Знаходимо PID процесу (process id)
ps aux | grep "[p]ython3 -m http.server"

# Або простіше:
kill %1
```

Тепер запускаємо його **правильно**, так, як повинен запускатися наш справжній OpenClaw агент на сервері — ізольовано від світу:

```bash
python3 -m http.server 18789 --bind 127.0.0.1 &
```

Перевірте `ss`:

```bash
sudo ss -tlnp | grep 18789
# Очікуваний результат: LISTEN ... 127.0.0.1:18789 ... python3
```

Перевірте зсередини VM:

```bash
curl localhost:18789
# Очікуваний результат: HTML код працює.
```

Ці дані 100% заблоковані від зовнішнього світу на рівні самої ВМ, незалежно від налаштувань роутерів чи Vagrant.

> 🛠 *Але тепер не працюватиме і наш браузер з хоста! В наступній Темі (SSH) ми навчимось створювати захищений зашифрований тунель безпосередньо в порт `127.0.0.1` нашої віртуальної машини, обходячи потребу у відкритих портах `0.0.0.0` назавжди.*

Зупиніть фоновий процес:

```bash
kill %1
```

### Крок 7: Фаєрвол UFW (Дозволено все, що працює)

Продемонструємо, як закрити доступ до відкритого (`0.0.0.0`) порту 80 за допомогою фаєрволу. Це потрібно для сервісів, які ми не можемо фізично прив'язати тільки до localhost, але хочемо контролювати IP.

```bash
# Запустимо "веб-сервер" на стандартному порту 80 (потребує sudo)
sudo python3 -m http.server 80 --directory /var/www/html/ &
```

Поки що фаєрвол вимкнено. Перевірте статус:

```bash
sudo ufw status
# Очікуваний результат: Status: inactive
```

Перевірте локально, що порт відповідає:

```bash
curl -I localhost:80
```

Увімкнемо фаєрвол. **КРИТИЧНА УВАГА: Якщо ви забудете першу команду (дозволити ssh), вас може назавжди відключити від сервера.**

```bash
# 1. ЗАВЖДИ спершу дозволяйте SSH!
sudo ufw allow 22/tcp

# 2. Увімкнути UFW
sudo ufw enable
# Підтвердіть клавішею 'y' (Command may disrupt existing ssh connections...)
```

Тепер, додамо і тоді заберемо правило:

```bash
# Дозволити доступ до порту веб-сервера (80)
sudo ufw allow 80/tcp
sudo ufw status

# Тепер заборонити доступ
sudo ufw deny 80/tcp
sudo ufw status
```

*Увага: Правило `deny` у нашому конкретному сетапі Vagrant->localhost може бути проігноровано через специфіку loopback iptables, але це класична поведінка на VPS з публічним IP. Перевіряючи доступ через публічний IP, ви отримаєте Time Out замість Connection Refused.*

Вимкніть UFW, щоб не зламати майбутні лабораторні:

```bash
sudo ufw disable
```

## ❓ Контрольні питання

1. Ви запустили сервіс бази даних на ВМ, але не можете підключитись до нього з вашого хост-комп'ютера за допомогою DBeaver чи іншого клієнта. Які два місця ви перевірите у першу чергу?
2. Що показує команда `ss -tlnp`? Для чого потрібен параметр `-p` і чому команда вимагає `sudo`?
3. Чому для запуску веб-сервера на порту 80 знадобилося використовувати команду `sudo python3 ...`, а для запуску на порту 18789 ми обійшлися без `sudo`?
4. Агент OpenClaw має конфігурацію: `HOST=127.0.0.1`, `PORT=18789`. Чому ми вибрали саме ці параметри для сервісу, який міститиме ваші чутливі фінансові дані, а не `HOST=0.0.0.0`?
5. Що обов'язково треба зробити ПЕРЕД виконанням команди `sudo ufw enable`, якщо ви адмініструєте сервер віддалено по SSH?
